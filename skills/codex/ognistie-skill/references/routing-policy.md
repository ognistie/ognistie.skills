# Routing policy

Apply the quality floor first and cost second. A cheaper model is acceptable only when it can complete the task reliably without likely retries or advanced review.

## Mandatory escalation

Select `advanced` for security, privacy, production, infrastructure, permissions, destructive operations, migrations, broad architecture, difficult diagnosis, high ambiguity, long-horizon work, or material financial, legal, safety, availability, or reputational risk.

## Economy

Use `economy` only for narrow, explicit, local, repetitive, reversible, low-risk work with clear acceptance criteria and inexpensive retries.

## Balanced

Use `balanced` for bounded implementation, debugging, UI work, API integration, test creation, or small refactors that trigger no advanced condition.

## Selection

1. Remove catalog candidates lacking required modalities, tools, context, output capacity, regional availability, or data-policy compatibility.
2. Estimate total expected cost, including likely retries and review.
3. Use price only as the final comparison among candidates meeting the quality floor.
4. If a promotional price expired, use the standard price until official documentation is checked.
5. If evidence is tied or the tier is uncertain, choose the stronger candidate.

Do not claim precise savings without measured token usage and current prices. Routing normally reduces cost per token, not token count.
